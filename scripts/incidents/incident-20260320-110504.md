## What happened
The eth0 interface was configured for DHCP only (ipv4.method=auto) with no static IP on the PLC subnet. The PLC (Click C0-10DD2E-D) lives at 169.168.10.21 on a direct Ethernet link. Without a 169.168.10.x address on eth0, there was no L2/L3 path to reach it.

The module's retry loop was firing every ~30s with TCP connect timeouts (105 consecutive failures at time of detection, ~52 minutes of downtime).

## What fixed it
Adding a secondary static IP (169.168.10.1/24) to eth0 restored connectivity immediately. The DHCP address (192.168.0.x) is preserved for internet access — both addresses coexist on the same interface.

The fix was made persistent via NetworkManager so it survives reboots.

## Open questions
- Why was the static IP never in the NM profile? Possible the original netplan config was removed or the Pi was reimaged without restoring static IP config.
- This fix assumes eth0 is physically connected to the PLC (or a switch with the PLC). The 0.258ms RTT confirms direct link.
- Should consider adding this to the fleet fragment or a provisioning script so new trucks don't hit the same issue.
