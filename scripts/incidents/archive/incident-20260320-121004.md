## Analysis
This is the 8th watchdog trigger today. The pattern shows two distinct connection failure episodes:

**Episode 1** (12:06:02–12:09:33): 7 failures over 3.5 minutes. All TCP connect timeouts. This is the tail end of the earlier eth0 NO-CARRIER recovery. The PLC's Modbus server may have been slow to accept connections after the extended Ethernet outage.

**Episode 2** (12:10:00–12:10:12): 3 failures over 12 seconds. Triggered by a bad register read at 12:09:58 — the PLC responded to TCP connect but returned 0 bytes for a DS register read. This suggests the Modbus TCP connection was established but the PLC's Modbus server wasn't fully ready to serve register reads. The module correctly detected the bad response, disconnected, and reconnected successfully 12 seconds later.

The fact that ICMP ping works perfectly (0.167ms, 0% loss) while Modbus TCP was flapping suggests this is a PLC-side Modbus server issue, not a network layer problem. The Click PLC's Modbus TCP stack may need a few seconds to stabilize after extended disconnection.
