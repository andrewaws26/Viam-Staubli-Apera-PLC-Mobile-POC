## Analysis
This is different from all previous incidents today. Earlier incidents (110504, 112501) involved software/configuration issues (missing static IP, transient DNS). This is a **physical layer failure** — the Ethernet PHY on the Pi detects no link partner.

Possible causes:
1. **Ethernet cable disconnected** — from Pi or PLC end (most likely if truck is in motion or someone bumped it)
2. **Ethernet cable damaged** — strain, vibration, or connector failure
3. **PLC powered off** — if the PLC lost power, its Ethernet port would go dark
4. **PLC Ethernet port failure** — less likely but possible
5. **Pi Ethernet port failure** — unlikely (would typically show other symptoms)

The brief internet "network is unreachable" errors at 11:54:21 may have been caused by NetworkManager briefly disrupting routing when it detected eth0 going down. Internet recovered by 11:54:24.

## What was tried
1. `ip link set eth0 down && up` — no effect, carrier still 0
2. `systemctl restart NetworkManager` — no effect, carrier still 0
3. Both are expected to fail for a physical link issue

## Required action (on-site)
1. Check that the Ethernet cable is firmly plugged into both the Pi's Ethernet port and the PLC's Ethernet port
2. Check that the PLC is powered on (PWR and RUN LEDs should be green)
3. Try a different Ethernet cable if available
4. Once carrier is restored, the PLC module will auto-reconnect (no restart needed)

## viam-server status
No restart needed. The PLC sensor module has built-in retry logic with 30s max backoff. Once eth0 link is restored and the static IP is assigned by NetworkManager, the next retry will reconnect automatically. viam-server itself is healthy — WebRTC and cloud connections are working over WiFi.
