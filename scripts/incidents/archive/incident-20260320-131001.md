## Diagnostics at 13:10
1. **viam-server**: active, PID 6679, uptime ~194s — running but cloud-disconnected
2. **Internet**: ping 8.8.8.8 → 56-77ms, 0% loss — healthy
3. **Viam Cloud ping**: app.viam.com → 38-96ms, 0% loss — reachable
4. **PLC ping**: 169.168.10.21 → 0.174ms, 0% loss — healthy
5. **eth0**: UP, 169.168.10.1/24 — connected to PLC
6. **wlan0**: UP, 192.168.0.176/24 on Verizon_X6JPH6 — internet working
7. **Capture**: .prog file at 200KB and growing — local data collection working
8. **Cloud sync**: FAILING — stuck in exponential backoff, "not connected"

## Error progression (from logs)
- 13:07:26 — "too many connection attempts; please wait a bit and try again" (rate limited)
- 13:07:52 — WebRTC connection state → disconnected
- 13:08:56 — "keepalive ping failed to receive ACK within timeout" (connection dead)
- 13:09:06 — "error reading server preface: EOF" (TLS/gRPC connection broken)
- 13:09:06 — data_manager enters exponential backoff (1m0s retry)
- 13:09:27 — "authentication handshake failed: not connected"
- 13:09:40+ — "DeadlineExceeded: not connected" (fully disconnected)

## This was NOT a false alarm
Unlike the previous 8+ false alarms today, this was a real issue. Internet was working but viam-server's gRPC session to Viam Cloud was stuck. Local capture continued fine but no data was reaching the cloud. Without restart, the exponential backoff would have delayed recovery further.

## Fix result
Restart at 13:11:00. By 13:11:04:
- Cloud connection re-established: "successfully established global connection to app"
- PLC reconnected in 1 second
- Capture collector reinitialized at 1Hz
- DNS 3/3, STUN 7/7 UDP + 2/2 TCP
- Robot constructed in 335ms

## Lesson
When internet is reachable but viam-server shows persistent "not connected" / "authentication handshake failed" errors, the gRPC session is stuck and a restart is the correct fix. The key diagnostic differentiator: ping to app.viam.com succeeds but viam-server logs show connection failures.
