## Root cause analysis (final)
After 16 false alarms and 13 repeated recommendations, I finally investigated the watchdog script itself. Found three compounding bugs:

### Bug 1: Wrong file extension (THE primary cause of all false alarms)
Line 53: `find ... -name "*.capture" -mmin -5`
Viam's data_manager writes `.prog` files while capturing. When a capture interval completes, it renames `.prog` → `.capture`, then syncs to cloud and DELETES the `.capture` file. So `.capture` files are transient and usually absent. The check should look for `.prog` files.

### Bug 2: No grace period after restart
When viam-server restarts (systemd auto-restart after panic), there's a 2-3 minute startup delay before capture begins. The watchdog's 5-minute window always overlaps this gap.

### Bug 3: No hysteresis
A single failed check immediately invokes Claude. Transient issues (network blip, file rotation timing) trigger expensive LLM calls.

## Fix details (on branch watchdog/fix-false-alarms)
1. Changed Check 4 to search for both `*.prog` AND `*.capture` files
2. Added secondary check: plate drop logs in journalctl prove system is active
3. Added 180-second grace period based on viam-server PID uptime
4. Added hysteresis: requires 2 consecutive failures before alerting Claude
5. Reset fail counter on all-checks-passed or grace-period skip
