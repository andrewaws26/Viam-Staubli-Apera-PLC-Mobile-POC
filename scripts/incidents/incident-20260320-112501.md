## Analysis
Two concurrent issues, likely related:

**PLC Modbus failures**: 10 consecutive TCP connect timeouts from ~11:20 to 11:25 (one every ~30s per retry interval). The PLC at 169.168.10.21 was unreachable despite eth0 having the correct 169.168.10.1/24 address. This was NOT the same root cause as incident-20260320-110504 (missing static IP) — the IP was correctly configured this time.

**DNS/internet failures**: Tailscale's MagicDNS resolver ([fd7a:115c:a1e0::53]) was returning "server misbehaving" for app.viam.com lookups. This caused: WebRTC signaling failures, data capture upload failures (exponential backoff to 1m), and network log appender disconnects.

The fact that both PLC (local Ethernet) and internet (WiFi/Tailscale) failed simultaneously suggests a broader network stack issue — possibly a brief NetworkManager hiccup, a WiFi disconnect/reconnect that temporarily disrupted eth0, or Tailscale DNS instability cascading to affect local routing.

The WebRTC connection at 11:24:30 succeeded (internet recovering), and the PLC reconnected at 11:25:03 (local recovering). Both self-healed within ~5 minutes.

## What fixed it
Nothing — both issues self-resolved. The PLC sensor module's retry loop (30s interval with exponential backoff) reconnected automatically. Viam's data_manager used exponential backoff for upload retries. The underlying network instability was transient.
